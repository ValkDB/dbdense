# Benchmark Workspace

This directory is for benchmark harness and scenarios only.
It is intentionally separate from production code paths.

## Non-Production Intent
- Files here should not be imported by `cmd/dbdense` or `internal/*` runtime packages.
- Keep benchmark code and assets isolated so normal `go test ./...` remains focused.
- Run artifacts are ignored in git via `benchmark/results/`.

## Goal
Produce reproducible evidence for accuracy and cost-transparency claims. The agentic MCP benchmark gives all arms the same Postgres MCP tool and natural-language business questions. The dbdense arms additionally receive precompiled schema context. Baseline must discover the schema through tool calls — that's where the token, latency, and accuracy delta becomes visible.

## Prerequisites
- Go 1.25
- Docker and Docker Compose (for seeded Postgres test database)
- Node.js / npx (for `@modelcontextprotocol/server-postgres` MCP server)
- Claude CLI installed and authenticated
- Compiled schema context files (see [Prepare Context Files](#prepare-context-files) below)

## Structure
- `scenarios/` benchmark prompts + expected behavior (7 scenarios)
- `mcp_postgres.json` MCP server config for Postgres tool access
- `datasets/` optional deterministic stress datasets (not loaded by default)
- `results/` per-run directories (`runs.jsonl`, `manifest.json`, `report.json`, `report.md`) (ignored)
- `benchmark/provider-flow.md` direct Claude CLI provider flow
- `benchmark/CHECKING.md` smoke-check/runbook for benchmark execution

## Execution Model
- Default run compares three arms:
  - **baseline**: Postgres MCP tool only (must discover schema via tool calls)
  - **dbdense**: Same Postgres MCP tool + precompiled schema context
  - **dbdense_lighthouse**: Same Postgres MCP tool + lighthouse map only
- All arms get the same system prompt, same natural-language questions, same MCP tool.
- The only difference: the dbdense arms receive schema context in the prompt.
- Capture real token usage, latency, turn count, and deterministic correctness.
- Correctness is response-based:
  - each scenario defines `verification.response.numeric_equals`
  - prompt uses natural business language (no column names, no SQL hints)
  - benchmark parses JSON from provider answer and verifies expected numeric fields

### Measured Fields (per run record)

| Field | What it is | Why it matters |
|-------|-----------|----------------|
| `usage.prompt_tokens` | Total input tokens the model processed, including cached portions (`input_tokens + cache_creation_input_tokens + cache_read_input_tokens` from the CLI). This is the **real context cost** per request. | In agentic mode, each tool-call turn re-sends conversation history. Baseline makes more turns for schema discovery, so its total input tokens should be higher. This is the definitive measure of context token cost. |
| `usage.completion_tokens` | Output tokens generated by the model. | Should be comparable across arms — the schema context helps the model answer correctly but shouldn't make it more verbose. |
| `usage.total_tokens` | `prompt_tokens + completion_tokens`. | Total token budget consumed. |
| `num_turns` | Number of agentic turns (tool-call round-trips). | Each turn re-sends the full conversation history. Baseline needs extra turns for schema discovery. Fewer turns = less context re-processing = lower total token cost. |
| `wall_latency_ms` | Wall-clock time from prompt submission to final answer. | Includes all tool-call round-trips. Directly correlates with turn count. |
| `estimated_input_tokens` | Local estimate of first-prompt tokens (`prompt_chars / 4`). | Legacy field. Less accurate than `usage.prompt_tokens` which includes all turns. Kept for backwards compatibility. |
| `accuracy.score` | Fraction of expected numeric fields that matched (0.0 to 1.0). | The primary metric — does the model get the right answer? |
| `accuracy.pass` | Whether all expected numeric fields matched exactly. | Binary pass/fail for the scenario. |

**Honest note on token reporting**: The Claude CLI reports `input_tokens`, `cache_creation_input_tokens`, and `cache_read_input_tokens` separately. We sum all three to get the real total context tokens processed. Earlier benchmark versions only captured the non-cached `input_tokens` (which is tiny), making input token comparisons meaningless. This is now fixed.

## Prepare Context Files
From repo root:

```bash
go run ./cmd/dbdense compile --in ctxexport.json --out benchmark/dbdense.schema.sql
go run ./cmd/dbdense compile --mode lighthouse --in ctxexport.json --out benchmark/dbdense.lighthouse.txt
```

## Optional: Add Deterministic 250-Table Noise Pack (Postgres)
Apply benchmark-only noise tables to the running Postgres container:

```bash
docker compose -f docker-compose.test.yml exec -T postgres \
  psql -U dbdense -d dbdense_test < benchmark/datasets/postgres_noise_250.sql
```

Verify exactly 250 tables were created:

```bash
docker compose -f docker-compose.test.yml exec -T postgres \
  psql -U dbdense -d dbdense_test -c \
  "SELECT count(*) FROM pg_catalog.pg_tables WHERE schemaname = 'bench_noise';"
```

## Run
### Default run (current profile)
Run this from the benchmark module for the current default profile:
- 1 run (smoke/proof)
- 3 arms (`baseline,dbdense,dbdense_lighthouse`)
- all scenarios in `benchmark/scenarios/` (regular + stress)

```bash
cd benchmark

go run ./cmd/benchrun \
  --provider claude \
  --model <model-id> \
  --claude-permission-mode bypassPermissions \
  --mcp-config-baseline "./mcp_postgres.json" \
  --mcp-config-dbdense "./mcp_postgres.json" \
  --mcp-config-dbdense-lighthouse "./mcp_postgres.json" \
  --context-file-dbdense "./dbdense.schema.sql" \
  --context-file-dbdense-lighthouse "./dbdense.lighthouse.txt"

go run ./cmd/benchreport --input ./results/<run_stamp> --target-arm dbdense
```

Before publishing performance claims, rerun with multiple passes:

```bash
cd benchmark

go run ./cmd/benchrun \
  --provider claude \
  --model <model-id> \
  --runs 3 \
  --query-timeout 5m \
  --claude-permission-mode bypassPermissions \
  --mcp-config-baseline "./mcp_postgres.json" \
  --mcp-config-dbdense "./mcp_postgres.json" \
  --mcp-config-dbdense-lighthouse "./mcp_postgres.json" \
  --context-file-dbdense "./dbdense.schema.sql" \
  --context-file-dbdense-lighthouse "./dbdense.lighthouse.txt"

go run ./cmd/benchreport --input ./results/<run_stamp> --target-arm dbdense
```
